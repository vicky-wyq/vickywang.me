JavaScript Learning





New Roadmap | Uses builder-loop to reinforce learning

Goal: Build scalable mental models + reusable code patterns
Principle: Every step builds both a small project and a deeper system insight
Style: Think first, build second, then refactor with higher clarity


Phase 0: Rethink Practice as System Creation
  • Start thinking of each JS practice project as a “unit of logic” for future reuse
  • For example, instead of "build a modal," think: → “Design a reusable visibility toggler + escape key handler”
  • Track learnings → Keep keywords: flow, trigger, state, side effect, DOM link, data layer


Phase 1: Core Mechanics (Refined Muscle Memory)
Objective: Deepen understanding of language fundamentals by building your own “toolkit”
Concept Mini Project System Thinking Angle
Variables & Types Budget Tracker Primitive vs Reference: memory, mutation
Functions & Scope Calculator Input/output pipeline, encapsulation
Arrays & Objects Grocery List Structure for state, iteration control
Loops & Logic Habit Tracker Flow control, trigger-condition-output
DOM Image Carousel UI as reactive data target
Events Modal Popup User intent as logic trigger
Tip: Always isolate 1 concept per file before merging.


Phase 2: Data + Async
Objective: Master state/data flow, async behavior, error-proof systems

Concept Mini Project System Lens
fetch() Weather App Input → API → parse → display → fail states
Promises Quote Generator Timing model, race conditions
Async/Await Currency Converter Controlled flow, fallback logic
LocalStorage To-Do App Persistence, sync UI ↔ storage
Practice: Draw “data journey” diagrams per feature:
API call → transform → update state → re-render → save if needed


Phase 3: Modular Design Thinking
Objective: Treat each feature as a “logic brick” that can snap into systems

Practice: Draw “data journey” diagrams per feature:
API call → transform → update state → re-render → save if needed

  • Rebuild to-do app using function modules (addTask, removeTask, syncState, updateDOM)
  • Try building a framework-less “Component” system:
    1. → Each module: state + DOM link + event logic
    2. → Reusable like React components, but pure JS
  • Explore Model–View–Control (MVC-lite) breakdowns



Phase 4: React Prep via Mental Translation
Objective: See React not as new, but as a formalization of what you already built

Concept What to Practice in JS Future in React
Props Parameterized modules Component inputs
State Central state object + render() useState()
Events Manual listeners + update logic JSX + onClick
Conditional rendering if blocks + DOM manipulation JSX conditionals
Componentization Functions returning UI chunks Functional components
🧠 You’re not memorizing React—you’re translating your logic model into its format.


Ongoing: System Thinking Logs

Track in Notion or Obsidian:
  • What logic was hard to follow? (e.g. async, bubbling, closure)
  • What worked better after refactor? Why?
  • What core patterns repeat? (e.g. loop → condition → output → store)

💬 Feedback Integration Loop

Every 2–3 mini projects:
  • Show to a dev friend or GPT
  • Ask: How would a React/TS engineer structure this?
  • Then rebuild to apply modular structure
